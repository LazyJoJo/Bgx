# Backend Change Propagation Workflow

> 通用需求变更工作流，适用于：新增功能、字段变更、类型修改、API调整、业务逻辑变更等所有后端变更场景。
> 核心理念：**自底向上修改，自顶向下验证**。

---

## 0. 需求理解

### 0.0 TDD核心理念（红-绿-重构循环）

> 本文档适用于TDD开发模式，遵循**测试优先**原则。

```
┌─────────────────────────────────────────────────────────┐
│                    TDD 循环                             │
│  ┌──────────┐    ┌──────────┐    ┌──────────┐          │
│  │   红     │───▶│   绿     │───▶│   重构   │──┐       │
│  │ 写失败测试│    │ 最小实现  │    │ 改善代码  │  │       │
│  └──────────┘    └──────────┘    └──────────┘  │       │
│       ▲────────────────────────────────────────┘       │
└─────────────────────────────────────────────────────────┘
```

**第1步：编写失败的测试（红）**
- 先写测试，明确期望行为
- 只写一个测试，聚焦当前功能点
- 运行测试，确认失败且失败原因正确（非编译错误）

**第2步：编写最简单的实现（绿）**
- 只写让测试通过的最小代码
- 允许重复、不完美，先让测试通过
- 不预判未来需求

**第3步：重构（保持绿）**
- 消除重复（DRY原则）
- 改善命名（见名知意）
- 运行测试确保重构未破坏功能

**第4步：循环**
- 下一测试用例或边界情况
- 直到功能完整覆盖

### 0.1 明确变更范围

在开始之前，必须清晰回答：

```
1. 涉及哪些业务实体？
2. 变更会影响哪些模块？
3. 是否跨模块共用同一实体？
4. 边界情况有哪些？
```

### 0.2 变更类型分类

```
□ A. 新增功能（新增表/实体/接口）
□ B. 字段变更（类型/新增/删除/默认值）
□ C. 业务逻辑变更（算法/流程调整）
□ D. API 变更（接口参数/返回值/协议）
□ E. 跨模块影响（多个模块共用同一实体）
```

### 0.3 初步影响评估

| 影响层级 | 检查要点 |
|---------|---------|
| 数据库 | 表结构、索引、约束、迁移 |
| Entity/DO/PO | 字段定义、类型、注解 |
| Repository | 查询方法、Mapper、SQL |
| Service | 方法签名、业务逻辑 |
| Domain Service | 领域规则、计算逻辑 |
| Controller | 参数校验、响应格式 |
| 前端/调用方 | 数据解析、展示格式 |

## 1. 变更传播路径

### 1.1 核心原则

```
变更传播方向：

Database ──► Entity ──► Repository ──► Service ──► Domain ──► Controller ──► 前端/调用方
   │           │            │            │            │            │
   │           │            │            │            │            │
   ▼           ▼            ▼            ▼            ▼            ▼
 约束/索引   字段定义     查询SQL      方法签名      领域规则     参数/响应
 类型精度    注解        resultMap    内部计算      阈值判断     格式转换
```

**修改顺序**：自底向上（Database → Entity → ... → Controller）
**验证顺序**：自顶向下（Controller → ... → Database）

### 1.2 各层检查清单

#### Database 层
```
□ 是否需要 CREATE TABLE / ALTER TABLE？
□ 是否有索引需求？（查询性能）
□ 是否有约束需求？（NOT NULL, UNIQUE, CHECK）
□ 数据迁移方案？
□ 回滚方案？
```

#### Entity 层
```
□ 字段类型是否与数据库匹配？
□ 注解是否正确？（@Column, @TableField, @Id）
□ 默认值是否合理？
□ 是否影响已有的 Builder / Constructor？
□ Lombok 注解是否正确？
```

#### Repository 层
```
□ Mapper XML 的 resultMap 是否与 Entity 一致？
□ 新增字段是否需要更新 SELECT/INSERT/UPDATE？
□ 复杂查询是否需要新增方法？
□ 是否有 N+1 查询风险？
```

#### Service 层
```
□ 方法签名是否需要更新？（参数类型、返回值类型）
□ 内部计算逻辑是否受影响？
□ 是否依赖该字段的其他服务？
□ 异常处理是否需要调整？
```

#### Domain Service 层
```
□ 领域逻辑是否受影响？
□ 计算规则是否需要修改？
□ 阈值判断逻辑是否需要调整？
□ 比较操作是否需要更新？（不同类型的比较语法不同）
```

#### Controller 层
```
□ 参数类型是否需要更新？
□ 响应 DTO 是否需要更新？
□ API 版本管理？（/v1/ → /v2/）
□ 是否向后兼容？
□ 参数校验规则是否需要调整？
```

#### 转换层（DTO ↔ Entity）
```
□ MapStruct/BeanUtils 映射是否正确？
□ 新增字段是否已添加到转换逻辑？
□ 是否有循环依赖风险？
□ 深拷贝 vs 浅拷贝选择是否正确？
□ 集合类型转换是否正确（List/Set/Map）？
□ 日期格式转换是否正确？
```

#### 前端/调用方
```
□ API 响应解析是否需要更新？
□ 展示格式是否需要调整？
□ 错误处理是否需要更新？
```

## 2. 缓存一致性检查

> 缓存是现代后端性能优化的重要手段，但字段变更时容易忽视缓存一致性。

### 2.1 缓存失效策略

```
□ 字段变更后，相关缓存是否清除？
□ 缓存Key是否包含变更的字段？
□ 是否有缓存穿透风险？（布隆过滤器/空值缓存）
□ 是否有缓存雪崩风险？（过期时间打散）
□ 缓存预热策略是否需要更新？
```

### 2.2 分布式缓存检查

```
□ 缓存更新是否使用原子操作？
□ 是否存在数据不一致窗口期？
□ 缓存与数据库的一致性策略？（Cache-Aside/Write-Through/Write-Behind）
□ 并发更新时的竞态条件处理？
```

### 2.3 各级缓存检查

| 缓存层级 | 检查要点 |
|---------|---------|
| 本地缓存（Caffeine） | 实例间一致性、过期时间 |
| 分布式缓存（Redis） | Key命名规范、序列化方式 |
| 应用缓存（Spring Cache） | @Cacheable/@CacheEvict 注解 |
| HTTP缓存 | ETag/Last-Modified 是否需要更新 |

## 3. 事务与并发控制

### 3.1 事务边界检查

```
□ 事务范围是否合理？（不过大也不过小）
□ 只读操作是否标记 @Transactional(readOnly=true)？
□ 跨服务调用是否在事务外？（防止长事务）
□ 分布式事务是否有补偿机制？（TCC/Saga）
□ 事务传播行为是否正确？（REQUIRED/REQUIRES_NEW）
```

### 3.2 并发安全检查

```
□ 是否有竞态条件？（Race Condition）
□ 是否需要乐观锁（@Version）？
□ 是否需要悲观锁（SELECT FOR UPDATE）？
□ 并发修改同一数据的处理？
□ 线程安全问题？（静态变量、单例bean）
```

### 3.3 隔离级别选择

| 场景 | 推荐隔离级别 | 说明 |
|-----|------------|------|
| 读多写少 | READ_COMMITTED | 默认，平衡性能与一致性 |
| 金融交易 | REPEATABLE_READ | 防止不可重复读 |
| 库存扣减 | SERIALIZABLE | 严格一致性，性能较差 |

## 4. 类型变更专项检查

### 2.1 类型传播检查

当修改字段类型时（如 String → Integer, Double → BigDecimal），必须检查：

```
□ Entity 层：字段类型已更新
□ Repository 层：resultMap 已同步
□ Service 层：方法签名已同步
□ Domain 层：计算逻辑已同步
□ Controller 层：参数/响应类型已同步
□ 前端/调用方：解析逻辑已同步
```

### 2.2 类型相关的语法差异

不同类型有不同的操作语法，修改时必须注意：

```java
// 数值类型比较
// Double: 直接比较
if (value > 0) { }

// BigDecimal: 必须使用 compareTo
if (value.compareTo(BigDecimal.ZERO) > 0) { }

// 字符串比较
// String: equals()
if (name.equals("ABC")) { }

// 枚举比较
if (status == OrderStatus.PENDING) { }  // 推荐
if (status.equals(OrderStatus.PENDING)) { }  // 也可
```

### 2.3 类型转换检查

```
□ 是否需要显式转换？（.doubleValue(), .intValue()）
□ 转换是否安全？（精度丢失风险）
□ null 值是否被正确处理？
□ 默认值是否合理？
```

## 5. API 兼容性策略

### 5.1 向后兼容性规则

**✅ 安全变更（向后兼容）：**
- 新增可选字段
- 新增接口端点
- 放宽校验规则
- 扩大返回数据范围
- 新增枚举值

**❌ 破坏性变更（需版本控制）：**
- 删除字段
- 修改字段类型
- 修改字段名
- 收紧校验规则
- 修改接口路径
- 修改业务逻辑结果

### 5.2 API 版本控制策略

```
方案A：路径版本控制（推荐）
  /api/v1/users
  /api/v2/users

方案B：媒体类型版本控制
  Accept: application/vnd.api.v1+json
  Accept: application/vnd.api.v2+json

方案C：查询参数版本控制
  /api/users?version=1
```

### 5.3 弃用策略

```
□ 旧API标记 @Deprecated 并说明替代方案
□ 维护文档记录弃用时间线和移除计划
□ 通知相关调用方（前端/第三方）
□ 监控旧API使用量
□ 设定最终移除日期（建议保留至少3-6个月）
```

## 6. 测试策略

### 3.1 测试覆盖矩阵

| 变更类型 | 单元测试 | 集成测试 | E2E 测试 |
|---------|---------|---------|---------|
| 新增功能 | ✅ 必须 | ✅ 必须 | ✅ 推荐 |
| 字段变更 | ✅ 必须 | ✅ 必须 | ✅ 推荐 |
| 类型修改 | ✅ 必须 | ✅ 必须 | ✅ 必须 |
| 业务逻辑 | ✅ 必须 | ✅ 必须 | ✅ 必须 |
| API 变更 | ✅ 必须 | ✅ 必须 | ✅ 必须 |
| Bug 修复 | ✅ 必须 | ✅ 推荐 | ⚠️ 视情况 |

### 3.2 单元测试要求

必须覆盖的场景：

```
1. Happy Path：正常业务流程
2. 边界值：该类型的最 小/大 值、空值
3. 异常处理：null 输入、空集合、非法参数
4. 类型转换：该类型与其他类型的转换正确性
```

### 3.3 Mock vs 真实组件

```
 Mock：隔离外部依赖，加快测试速度
 真实：验证真实集成，发现 Mock 无法发现的问题

 原则：
 - Repository → 集成测试用真实，单元测试用 Mock
 - Service → 集成测试用真实
 - Controller → @WebMvcTest 或 Testcontainers
 - ⚠️ 关键路径必须使用真实组件
```

### 3.4 测试数据规范

```
❌ 禁止：假数据（如 1.0, 2.0, "test"）
✅ 要求：使用业务范围内真实数据
   - 字符串：有意义的业务值
   - 数值：符合业务范围的真实值
   - 日期：符合业务场景的真实日期
```

## 7. 编译与验证

### 4.1 必须执行的验证

| 顺序 | 命令 | 目的 |
|-----|------|------|
| 1 | `mvn clean compile` | 验证所有源码编译通过 |
| 2 | `mvn test-compile` | 验证测试代码编译通过 |
| 3 | `mvn test` | 运行单元测试 |
| 4 | `mvn verify` | 运行集成测试 |

### 4.2 常见编译错误

```
1. Type mismatch
   → 类型传播不完整，上下 游类型不一致
   → 从 Entity 向上游逐层检查

2. Method not found
   → 方法签名变更后引用位置未同步
   → 使用 "Find Usages" 查找所有引用

3. NullPointerException
   → null 值传播
   → 检查赋值和使用路径

4. ClassCastException
   → 尝试将 A 类型强制转换为 B 类型
   → 使用正确的转换方法
```

### 4.3 何时必须 clean build

```
□ 修改了字段类型时
□ 修改了泛型参数时
□ 修改了编译配置时
□ 遇到奇怪的编译错误时
□ CI/CD / 部署环境
```

## 8. 代码审查

### 5.1 变更完整性检查

```
□ 所有使用该字段/方法的位置都已被识别
□ 变更已传播到所有受影响的上游
□ 类型相关的语法差异已处理
□ 测试用例覆盖了边界值和异常情况
```

### 5.2 安全检查

```
□ 无 SQL 注入风险（参数化查询）
□ 无硬编码凭据/密钥
□ 用户输入已校验
□ 错误信息不泄露敏感信息
```

### 5.3 性能检查

```
□ 复杂查询有适当索引
□ 无 N+1 查询问题
□ 无不必要的全表扫描
```

## 附录：常见问题

### Q1: 如何快速定位传播遗漏？

```
1. 从变更点开始，使用 IDE "Find Usages" 逐层向上检查
2. 路径：Entity → Repository → Service → Domain → Controller
3. 特别关注：方法签名、比较操作、算术运算
4. 执行 `mvn clean compile` 强制重新编译
```

### Q2: 何时需要 E2E 测试？

```
□ 是否是核心业务流程？
□ 是否涉及多个系统/模块交互？
□ 是否有用户直接操作路径？
```

### Q3: 如何避免遗漏？

```
1. 变更开始前：画出演播路径图
2. 变更过程中：逐层检查清单
3. 变更完成后：执行完整编译和测试
4. 审查阶段：对照检查清单验证
```

### Q4: 如何高效使用IDE进行变更影响分析？

```
1. 定位变更起点（Entity/Field）
2. 使用 Find Usages (Alt+F7) 查找所有引用
3. 使用 Call Hierarchy (Ctrl+Alt+H) 查看调用链
4. 使用 Change Signature (Ctrl+F6) 安全修改方法签名
5. 使用 Inspect Code 进行静态分析
6. 使用 Rename (Shift+F6) 安全重命名
```

### Q5: 如何处理跨多个模块的变更？

```
1. 识别共享的实体/DTO
2. 使用 IDE 的 Find Usages 跨模块搜索
3. 优先修改被依赖的模块（自底向上）
4. 使用 Maven/Gradle 的依赖分析工具
5. 确保所有模块的测试都通过
6. 协调各模块的发布顺序
```
