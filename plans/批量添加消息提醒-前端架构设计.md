### 5.3 防抖封装

```typescript
// app/src/utils/debounce.ts

export function debounce<T extends (...args: any[]) => any>(
  func: T,
  wait: number
): (...args: Parameters<T>) => void {
  let timeout: NodeJS.Timeout | null = null
  
  return function(...args: Parameters<T>) {
    if (timeout) clearTimeout(timeout)
    timeout = setTimeout(() => func(...args), wait)
  }
}

// 使用示例
const debouncedSearch = debounce((keyword: string, type: string) => {
  searchSymbols(keyword, type)
}, 300)
```

## 6. 表单验证策略

### 6.1 验证规则

| 验证项 | 规则 | 错误消息 | 触发时机 |
|-------|------|---------|---------|
| 标的类型 | 必填 | "请先选择标的类型" | 提交时 |
| 标的选择 | 至少1个，最多100个 | "请至少选择一个标的" / "单次最多选择100个标的" | 提交时/选择时 |
| 提醒类型 | 必填 | "请选择提醒类型" | 提交时 |
| 目标价格 | 价格类型时必填，>0，2位小数 | "请输入有效价格" | 输入时/提交时 |
| 涨跌幅 | 涨跌幅类型时必填，≠0，范围[-99, 99] | "请输入有效涨跌幅" | 输入时/提交时 |

### 6.2 Ant Design Form验证示例

```typescript
const validateSymbols = (_: any, value: string[]) => {
  if (!value || value.length === 0) {
    return Promise.reject(new Error('请至少选择一个标的'))
  }
  if (value.length > 100) {
    return Promise.reject(new Error('单次最多选择100个标的'))
  }
  return Promise.resolve()
}

const validateTargetPrice = (_: any, value: number) => {
  const alertType = form.getFieldValue('alertType')
  
  if (alertType !== 'PERCENTAGE_CHANGE') {
    if (value === undefined || value === null) {
      return Promise.reject(new Error('请输入目标价格'))
    }
    if (value <= 0) {
      return Promise.reject(new Error('价格必须大于0'))
    }
  }
  return Promise.resolve()
}

const validateTargetChangePercent = (_: any, value: number) => {
  const alertType = form.getFieldValue('alertType')
  
  if (alertType === 'PERCENTAGE_CHANGE') {
    if (value === undefined || value === null) {
      return Promise.reject(new Error('请输入目标涨跌幅'))
    }
    if (value === 0) {
      return Promise.reject(new Error('涨跌幅不能为0'))
    }
    if (Math.abs(value) > 99) {
      return Promise.reject(new Error('涨跌幅超出合理范围'))
    }
  }
  return Promise.resolve()
}
```

### 8.2 关键交互说明

#### 8.2.1 搜索防抖
- 用户输入关键词后，延迟300ms调用搜索API
- 避免每次按键都触发请求

#### 8.2.2 重复检测
- **方案A**: 提交前调用`POST /alerts/check-duplicates`
- **方案B**: 提交后由后端检测，前端处理部分成功响应
- **推荐**: 方案B(减少一次API调用，后端保证一致性)

#### 8.2.3 部分成功处理
```typescript
// 处理批量创建响应
const handleBatchCreateResult = (result: BatchCreateResult) => {
  if (result.successCount === result.totalCount) {
    // 全部成功
    message.success(`成功创建 ${result.successCount} 条提醒`)
    navigate('/alerts')
  } else if (result.successCount > 0) {
    // 部分成功
    setResultModalVisible(true)
    setBatchResult(result)
  } else {
    // 全部失败
    setResultModalVisible(true)
    setBatchResult(result)
  }
}
```

## 9. 关键技术决策

### 9.1 类型前置 vs 类型内联

| 方案 | 优点 | 缺点 |
|------|------|------|
| **类型前置(推荐)** | 逻辑清晰，避免混淆股票和基金 | 多一步操作 |
| 类型内联 | 操作更少 | 可能混选股票和基金，增加复杂性 |

**决策**: 采用类型前置方案，符合PRD要求。

### 9.2 重复检测时机

| 方案 | 优点 | 缺点 |
|------|------|------|
| **提交后检测(推荐)** | 减少API调用，后端保证一致性 | 用户需等待完整结果 |
| 提交前检测 | 用户提前知道重复项 | 增加一次API调用，可能存在竞态 |

**决策**: 采用提交后检测，由后端返回部分成功响应。

### 9.3 状态管理方案

| 方案 | 适用场景 | 选择 |
|------|---------|------|
| **Redux Toolkit** | 跨组件共享状态 | ✅ 项目已有 |
| React Context | 简单全局状态 | 不适用 |
| 本地State | 组件内部状态 | 配合使用 |

**决策**: 继续使用Redux Toolkit管理批量创建状态，组件内部状态使用useState。

### 9.4 搜索实现

| 方案 | 优点 | 缺点 |
|------|------|------|
| **Select mode="multiple" + onSearch(推荐)** | Ant Design原生支持 | 大数据量可能性能问题 |
| 自定义搜索+复选框 | 更灵活 | 开发成本高 |

**决策**: 使用Ant Design Select组件，增加防抖优化。
