## 4. 批量创建提醒核心业务流程设计

### 4.1 现有流程分析

```java
// 现有AlertAppServiceImpl.batchCreateAlert()流程:

for (String symbol : request.getSymbols()) {
    try {
        // 1. 构造单条创建请求
        CreateAlertRequest singleRequest = ...;
        
        // 2. 调用单条创建方法(内部有重复检测)
        CreateAlertResponse response = createAlert(singleRequest);
        
        // 3. 判断是否新创建或已存在
        if (response.isCreated()) {
            successList.add(response.getAlert());
        } else {
            successList.add(response.getAlert());  // 已存在也算成功
        }
    } catch (Exception e) {
        failList.add(FailedAlert.builder()
            .symbol(symbol)
            .reason(e.getMessage())
            .build());
    }
}
```

### 4.2 现有流程的问题

| 问题 | 影响 | 严重性 |
|------|------|-------|
| 循环调用单条创建 | N次数据库查询,性能差 | 高 |
| 已存在也算成功 | 前端无法区分新创建和已存在 | 中 |
| 缺少batchId | 无法追踪批次 | 低 |
| 无批量插入优化 | 未利用MyBatis batch insert | 高 |
| 失败原因不够具体 | 只返回异常消息,无错误码 | 中 |

### 4.3 优化后的流程设计

```java
@Override
@Transactional(readOnly = true)  // 只读事务,用于批量查询
public BatchCreateAlertResponse batchCreateAlert(BatchCreateAlertRequest request) {
    logger.info("批量创建提醒: 用户ID={}, 标的数量={}, 类型={}",
            request.getUserId(), request.getSymbols().size(), request.getSymbolType());

    // 1. 参数校验
    validateBatchRequest(request);

    // 2. 批量查询已存在的提醒(一次查询代替N次)
    List<PriceAlert> existingAlerts = priceAlertRepository
            .findByUserIdAndSymbolsAndSymbolTypeAndAlertType(
                request.getUserId(),
                request.getSymbols(),
                request.getSymbolType(),
                request.getAlertType()
            );
    
    Set<String> existingSymbols = existingAlerts.stream()
            .map(PriceAlert::getSymbol)
            .collect(Collectors.toSet());

    // 3. 过滤出需要创建的标的
    List<String> symbolsToCreate = request.getSymbols().stream()
            .filter(symbol -> !existingSymbols.contains(symbol))
            .collect(Collectors.toList());

    // 4. 构建批量插入的实体列表
    List<PriceAlert> alertsToCreate = symbolsToCreate.stream()
            .map(symbol -> buildPriceAlert(request, symbol))
            .collect(Collectors.toList());

    // 5. 批量插入(开启事务)
    List<PriceAlert> createdAlerts = batchInsertAlerts(alertsToCreate);

    // 6. 构建响应
    List<SuccessAlert> successList = buildSuccessList(createdAlerts, existingAlerts);
    List<FailedAlert> failList = buildFailureList(request.getSymbols(), createdAlerts, existingAlerts);

    return BatchCreateAlertResponse.builder()
            .batchId(generateBatchId())
            .totalCount(request.getSymbols().size())
            .successCount(successList.size())
            .failureCount(failList.size())
            .successList(successList)
            .failureList(failList)
            .build();
}
```

### 4.4 优化点说明

| 优化项 | 优化前 | 优化后 | 性能提升 |
|-------|-------|-------|---------|
| 重复检测 | N次单独查询 | 1次批量IN查询 | ~90% |
| 插入方式 | N次单独insert | 1次batch insert | ~80% |
| 事务控制 | 无统一事务 | 只读事务查询 + 写事务插入 | 数据一致性 |

## 5. 事务边界和并发控制策略

### 5.1 事务设计

```java
// 查询阶段: 只读事务
@Transactional(readOnly = true)
public BatchCreateAlertResponse batchCreateAlert(BatchCreateAlertRequest request) {
    // 1. 批量查询已存在的提醒
    // 2. 过滤需要创建的标的
    // 3. 调用批量插入方法
    return batchInsertInNewTransaction(request, symbolsToCreate);
}

// 插入阶段: 写事务
@Transactional(propagation = Propagation.REQUIRES_NEW)
private List<PriceAlert> batchInsertInNewTransaction(
        BatchCreateAlertRequest request, 
        List<String> symbolsToCreate) {
    // 批量插入
    return priceAlertRepository.saveAll(alertsToCreate);
}
```

### 5.2 并发控制

#### 5.2.1 数据库层面
```sql
-- price_alert表唯一约束(防止并发重复创建)
ALTER TABLE price_alert 
ADD UNIQUE INDEX uk_user_symbol_type_alert 
(user_id, symbol, symbol_type, alert_type);
```

#### 5.2.2 应用层面
```java
// 乐观锁: 插入时捕获DuplicateKeyException
try {
    priceAlertMapper.batchInsert(alertsToCreate);
} catch (DuplicateKeyException e) {
    // 并发导致重复,查询已存在的记录
    List<PriceAlert> existing = priceAlertRepository
            .findByUserIdAndSymbolsAndSymbolTypeAndAlertType(...);
    // 返回已有记录作为成功结果
}
```

### 5.3 并发场景处理

| 场景 | 处理方式 | 结果 |
|------|---------|------|
| 同用户同时批量创建相同标的 | 唯一约束 + DuplicateKeyException捕获 | 一个成功,另一个返回已存在 |
| 快速重复提交(网络重试) | 幂等键机制(如需要) | 返回相同响应 |
| 插入过程中其他用户操作 | 无影响,用户级别隔离 | 正常处理 |

## 7. 部分失败场景处理策略

### 7.1 失败场景分类

| 失败类型 | 错误码 | 处理策略 | 可重试 |
|---------|-------|---------|-------|
| 标的已存在 | ALERT_DUPLICATE | 跳过,返回已存在记录 | 否 |
| 标的不存在 | TARGET_NOT_FOUND | 记录失败,继续处理其他 | 否 |
| 数据异常 | DATA_ERROR | 记录失败,继续处理其他 | 视情况 |
| 数据库错误 | DB_ERROR | 回滚整个批次 | 是 |
| 数量超限 | LIMIT_EXCEEDED | 停止处理,返回已成功部分 | 否 |

### 7.2 失败处理策略

```java
// 策略1: 单个标的失败不影响其他(推荐)
for (String symbol : symbols) {
    try {
        // 创建逻辑
        successList.add(result);
    } catch (BusinessException e) {
        // 业务异常: 记录失败,继续处理
        failList.add(FailedAlert.builder()
            .symbol(symbol)
            .reason(e.getMessage())
            .errorCode(e.getErrorCode())
            .build());
    } catch (Exception e) {
        // 系统异常: 记录日志,继续处理
        logger.error("创建提醒失败: symbol={}", symbol, e);
        failList.add(FailedAlert.builder()
            .symbol(symbol)
            .reason("系统异常")
            .errorCode("SYSTEM_ERROR")
            .build());
    }
}

// 策略2: 数据库异常回滚整个批次
@Transactional(propagation = Propagation.REQUIRES_NEW)
public List<PriceAlert> batchInsertAlerts(List<PriceAlert> alerts) {
    if (alerts.isEmpty()) {
        return Collections.emptyList();
    }
    try {
        return priceAlertRepository.saveAll(alerts);
    } catch (DataAccessException e) {
        // 数据库异常,回滚
        throw new BatchInsertException("批量插入失败", e);
    }
}
```

## 8. 幂等性设计

### 8.1 幂等键机制(可选)

```java
// 请求头传递幂等键
@PostMapping("/batch")
public ApiResponse<BatchCreateAlertResponse> batchCreateAlert(
        @RequestHeader(value = "Idempotency-Key", required = false) String idempotencyKey,
        @Valid @RequestBody BatchCreateAlertRequest request) {
    
    if (idempotencyKey != null) {
        // 检查是否已处理过此幂等键
        Optional<BatchCreateAlertResponse> cached = idempotentCache.get(idempotencyKey);
        if (cached.isPresent()) {
            return ApiResponse.success("幂等请求,返回缓存结果", cached.get());
        }
    }
    
    BatchCreateAlertResponse response = alertAppService.batchCreateAlert(request);
    
    if (idempotencyKey != null) {
        // 缓存响应结果,24小时过期
        idempotentCache.put(idempotencyKey, response, 24, TimeUnit.HOURS);
    }
    
    return ApiResponse.success("批量创建完成", response);
}
```

### 8.2 自然幂等性

由于`price_alert`表有唯一约束`uk_user_symbol_type_alert`,同一用户对同一标的的相同类型提醒只能创建一次,因此批量创建接口具有**自然幂等性**:

- 重复提交相同请求 → 部分/全部返回"已存在"
- 不会产生重复数据

## 9. 性能优化策略

### 9.1 批量插入优化

```java
// MyBatis Plus批量插入
@Service
public class PriceAlertRepositoryImpl implements PriceAlertRepository {
    
    @Autowired
    private PriceAlertMapper priceAlertMapper;
    
    @Override
    @Transactional
    public List<PriceAlert> saveAll(List<PriceAlert> alerts) {
        if (alerts == null || alerts.isEmpty()) {
            return Collections.emptyList();
        }
        
        // 方式1: MyBatis Plus saveBatch (默认每批1000条)
        List<PriceAlertPO> poList = alerts.stream()
                .map(this::convertToPO)
                .collect(Collectors.toList());
        
        // 使用MyBatis Plus的批量插入
        boolean success = this.saveBatch(poList);
        if (!success) {
            throw new RuntimeException("批量插入失败");
        }
        
        return poList.stream()
                .map(this::convertToEntity)
                .collect(Collectors.toList());
    }
}
```
