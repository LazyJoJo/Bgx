# 项目开发与验收规范

> 基于基金添加功能踩坑复盘

## 一、本次 Bug 根因分析

### 背景

基金添加功能的 Playwright 自动化测试 20/20 全部通过，但用户手动测试时发现：输入一个后端不存在的基金代码（如 `014143`）后，"添加"按钮无法点击。

### 根因分析

#### 1. 测试用例设计偏向"黄金路径"（Happy Path）

- 所有测试都使用真实存在的基金代码 `001593`
- 覆盖了：空输入禁用、有效输入启用、成功添加
- 但**没有任何测试用例验证**：当输入一个不存在的基金代码时，按钮是否可以点击
- 缺少"反向用例"（Negative Cases）

#### 2. 验收标准只验证"符合 PRD"，不验证"用户可用"

- 验收报告检查的是 PRD 功能点是否实现
- 但 PRD 本身没有明确写出边界条件
- 导致验收流于"功能点打勾"，而非"用户体验验证"

#### 3. 按钮状态与异步查询结果错误耦合

- 原代码：`disabled={!fundInfo}`
- 按钮禁用条件依赖后端异步查询结果
- 查询失败时 `fundInfo` 为 null，按钮永远禁用
- **前端最佳实践**：按钮状态只应由本地可验证规则决定

#### 4. AI 生成代码和测试的系统性偏差

- AI 生成代码时默认假设"用户输入有效数据"
- 生成的测试用例也遵循同样假设
- 缺乏"故意输入错误数据来验证系统行为"的思维

---

## 二、测试用例编写规范（核心）

每个涉及用户输入的功能，测试用例**必须**覆盖以下三类场景：

| 场景类型 | 说明 | 示例 |
|---------|------|------|
| 前端格式校验 | 空值、格式错误、长度不足 | 输入少于6位基金代码 |
| 后端业务校验 | 格式正确但业务无效 | 输入6位但数据库不存在的基金代码 |
| 成功路径 | 格式正确且业务有效 | 输入真实存在的基金代码并添加成功 |

### 数据准备规范

- **禁止**使用硬编码的猜测数据（如 `000001`）
- 测试前必须通过真实 API 查询获取有效数据
- 测试依赖的环境数据应在 `beforeAll` 中通过 API 动态获取并记录

### Playwright 选择器规范

- 优先使用 `data-testid` 定位元素
- **禁止**基于按钮文本的模糊匹配（如 `getByText('添加')`）
- 必须读取源码确认真实的 UI 文本和结构

### 单步调试原则

- **禁止**使用 `--grep` 批量运行多个测试
- 必须逐个运行测试用例
- 出现一个 fail 立即停止，分析根因后再继续

---

## 三、前端交互设计规范

### 按钮禁用逻辑原则

- 禁用条件只能是**本地可验证的规则**（如格式校验、必填校验）
- **禁止**将按钮禁用状态与异步 API 返回结果绑定
- 后端业务校验结果应通过错误提示展示，而不是通过禁用按钮阻止用户操作

### 错误状态管理

用户输入后触发查询，查询失败时应：

1. 显示明确的错误提示（如"基金代码不存在"）
2. 保持按钮可点击，允许用户修改后重试
3. **不要**进入"死状态"（无任何提示且无法操作）

### 异步操作状态

- **加载中**：显示 loading 状态，但按钮保持可点击或显示"加载中"文案
- **成功**：正常流转
- **失败**：显示错误信息，允许重试

---

## 四、验收流程规范

自动化测试通过后，**必须**执行以下验收步骤：

### 1. 边界条件检查表（每个功能必须勾选）

- [ ] 输入无效格式数据，前端是否正确拦截？
- [ ] 输入格式正确但业务无效数据，后端是否返回明确错误？
- [ ] 后端返回错误时，前端是否进入"死状态"？
- [ ] 重复提交时，系统是否幂等？
- [ ] 网络超时或服务不可用时，前端如何降级？

### 2. 探索性测试（15分钟）

- 故意输入"预期外"的数据
- 观察系统行为是否符合直觉
- **关键问题**：如果后端 API 返回错误，前端 UI 是否仍然可用？

### 3. Code Review 增加"异常路径"检查

- 该功能是否处理了至少 3 种错误情况？
- 按钮禁用逻辑是否合理？
- 错误提示是否明确？

---

## 五、PRD 评审检查清单

在需求评审阶段，**必须**明确以下边界：

- [ ] 用户输入无效数据时，系统如何响应？
- [ ] 后端服务不可用时，前端如何降级？
- [ ] 重复操作时，系统是否幂等？
- [ ] 异步操作超时如何处理？
- [ ] 是否存在"死状态"（用户无法继续操作且无任何提示）？

---

## 六、AI 辅助开发审查要点

当使用 AI 生成代码和测试时，人工**必须**重点审查：

### 1. 代码审查

- 检查所有 `disabled` 属性，确认**不依赖异步状态**
- 检查所有 `catch` 块，确认错误后有明确的 UI 反馈
- 检查是否处理了"加载中"、"成功"、"失败"三种状态

### 2. 测试审查

- 检查测试数据是否来自真实 API，而非硬编码猜测
- 检查是否包含**至少一个反向用例**
- 检查选择器是否基于 `data-testid` 而非文本

### 3. 验收审查

- **不能**仅因"自动化测试通过"就标记验收完成
- 必须完成探索性测试和边界条件检查表

## 附录：常见错误模式

| 错误模式 | 问题 | 正确做法 |
|---------|------|---------|
| `disabled={!apiResult}` | API 失败时按钮永久禁用 | `disabled={!inputValid}` 本地校验 |
| 仅有成功路径测试 | 遗漏错误场景 | 覆盖正向+反向+异常三类 |
| 硬编码测试数据 | 环境不一致时测试失败 | 通过 API 动态获取测试数据 |
| 使用文本选择器 | UI 文本变更导致测试失败 | 使用 `data-testid` |
