## 1. 技术方案

### 1.1 交互设计

采用**单页切换式**（方案B）：顶部两个大按钮切换快速模式/自定义模式。

**设计原则：**
- 模式切换器置于页面顶部，作为第一个视觉焦点
- 快速模式表单极简，自定义模式表单分区清晰
- 组件间距统一，呼吸感强
- 适当使用图标、色彩点缀

## 3. 数据库设计

### 3.1 字段命名

> **原则**：数据库字段尽量不改变，代码层直接使用数据库字段名

**统一字段命名**：

| 字段名 | 说明 | 备注 |
|-------|------|------|
| `symbol` | 标的代码 | 代码层直接使用此字段名 |
| `symbol_type` | 标的类型 | 代码层直接使用此字段名，值：STOCK/FUND |
| `symbol_name` | 标的名称 | 代码层直接使用此字段名 |
| `target_change_percent` | 涨跌幅阈值 | 代码层直接使用此字段名 |

**说明**：
- 代码层直接使用数据库字段名（symbol, symbol_type, symbol_name, target_change_percent）
- 代码层变量名与数据库字段名保持一致（驼峰命名：symbol, symbolType, symbolName, targetChangePercent）

### 3.2 表作用区分

> **重要**：本模块涉及两个核心表，以下是它们的职责区分：

| 表名 | 职责 | 产生方式 |
|------|------|----------|
| **user_subscription** | 用户订阅的**标的和阈值设置** | 用户手动订阅（新建表） |
| **risk_alert** | 系统定时检测生成的**风险记录** | 11:30/14:30自动创建（已有表） |
| ~~price_alert~~ | ~~已废弃~~ | ~~用 user_subscription 替代~~ |
| ~~alert_history~~ | ~~已废弃~~ | ~~不再需要~~ |

### 3.3 user_subscription 表（新建表）

**字段设计**：

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGSERIAL | 主键 |
| user_id | BIGINT | 用户ID |
| symbol | VARCHAR(20) | 标的代码 |
| symbol_type | VARCHAR(20) | 标的类型：STOCK/FUND |
| symbol_name | VARCHAR(100) | 标的名称 |
| target_change_percent | DECIMAL(10,2) | 目标涨跌幅阈值（如：1.00表示1%） |
| status | VARCHAR(20) | 订阅状态：ACTIVE/INACTIVE |
| created_at | TIMESTAMP | 创建时间 |
| updated_at | TIMESTAMP | 更新时间 |

```sql
CREATE TABLE IF NOT EXISTS user_subscription (
    id BIGSERIAL PRIMARY KEY,
    user_id BIGINT NOT NULL,
    symbol VARCHAR(20) NOT NULL,                    -- 标的代码
    symbol_type VARCHAR(20) NOT NULL,               -- 标的类型：STOCK/FUND
    symbol_name VARCHAR(100),                        -- 标的名称
    target_change_percent DECIMAL(10,2) NOT NULL,    -- 目标涨跌幅阈值（如：1.00表示1%）
    status VARCHAR(20) DEFAULT 'ACTIVE',             -- 订阅状态：ACTIVE/INACTIVE
    created_at TIMESTAMP WITHOUT TIME ZONE DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP WITHOUT TIME ZONE DEFAULT CURRENT_TIMESTAMP,
    -- 唯一约束：同一用户同一标的只能有一个活跃订阅
    CONSTRAINT uq_user_subscription UNIQUE (user_id, symbol, symbol_type)
);

-- 索引
CREATE INDEX IF NOT EXISTS idx_user_subscription_user ON user_subscription(user_id);
CREATE INDEX IF NOT EXISTS idx_user_subscription_status ON user_subscription(status) WHERE status = 'ACTIVE';
CREATE INDEX IF NOT EXISTS idx_user_subscription_entity ON user_subscription(symbol, symbol_type);
```

### 3.4 risk_alert 表（已有表）

> **重要**：此表是**已有表**，数据库已存在。用于存储每次风险检测的结果记录。

**数据库实际字段**：symbol, symbol_type, symbol_name（与文档中描述的 entity_code 等不同）

```sql
-- 此表为已有表，以下为参考结构（实际以数据库现有字段为准）
CREATE TABLE IF NOT EXISTS risk_alert (
    id BIGSERIAL PRIMARY KEY,
    user_id BIGINT NOT NULL,
    symbol VARCHAR(20) NOT NULL,                    -- 标的代码
    symbol_type VARCHAR(20) NOT NULL,                -- 标的类型：STOCK/FUND
    symbol_name VARCHAR(100),                       -- 标的名称
    alert_date DATE NOT NULL,                       -- 风险日期
    time_point VARCHAR(10) NOT NULL,                -- 时间点：11:30 / 14:30
    has_risk BOOLEAN DEFAULT FALSE,                 -- 是否触发风险
    change_percent DECIMAL(10,2),                    -- 涨跌幅（2位小数）
    current_price DECIMAL(10,2),                    -- 当前价格（2位小数）
    yesterday_close DECIMAL(10,2),                   -- 昨日收盘价（2位小数）
    is_read BOOLEAN DEFAULT FALSE,                   -- 是否已读
    triggered_at TIMESTAMP WITHOUT TIME ZONE,       -- 触发时间
    created_at TIMESTAMP WITHOUT TIME ZONE DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP WITHOUT TIME ZONE DEFAULT CURRENT_TIMESTAMP,
    -- 唯一约束：同一用户同一标的同一日期同一时间点只有一条记录
    CONSTRAINT uq_risk_alert UNIQUE (user_id, symbol, alert_date, time_point)
);

-- 索引
CREATE INDEX IF NOT EXISTS idx_risk_alert_user ON risk_alert(user_id);
CREATE INDEX IF NOT EXISTS idx_risk_alert_date ON risk_alert(alert_date);
CREATE INDEX IF NOT EXISTS idx_risk_alert_user_date ON risk_alert(user_id, alert_date);
CREATE INDEX IF NOT EXISTS idx_risk_alert_unread ON risk_alert(user_id, is_read) WHERE is_read = FALSE;
```

- VOLUME_SPIKE - 成交量异常

## 8. 功能移除

> **移除日期**：2026-04-15
> **移除原因**：功能与 user_subscription/risk_alert 重复，简化数据模型
> **重要说明**：本次移除为**物理删除**，不是保留废弃代码。所有涉及 price_alert 和 alert_history 的代码必须从代码库中彻底移除。

### 8.1 移除范围

| 移除项 | 说明 | 替代方案 |
|--------|------|---------|
| price_alert 表 | 用户主动创建的价格监控规则表 | 使用 user_subscription 表 |
| alert_history 表 | price_alert 触发后的历史记录表 | 直接使用 risk_alert 表 |
| price_alert 相关 API | 创建/查询/编辑/删除提醒的接口 | 使用 subscription 相关 API |
| alert_history 相关 API | 查询历史记录的接口 | 不再需要，风险提醒直接查看 risk_alert |

### 8.2 移除方式

**必须物理删除以下代码，不能仅标记为废弃：**
- 删除整个 Controller（如果只包含废弃接口）
- 删除整个 Service（如果只包含废弃逻辑）
- 删除整个 Mapper/Repository（如果只用于废弃表）
- 删除整个前端页面组件（如果只用于废弃功能）
- 删除废弃的 API 服务文件
- 删除废弃的 Redux slice

**不能仅做"注释掉"或"标记为废弃"处理**，必须完全移除以避免代码污染和维护负担。

### 8.2 移除后的数据流向

**移除前（原有架构）**：
```
用户创建 price_alert → 定时任务检测 → 满足条件时触发
                                      ↓
                              创建 alert_history（历史记录）
                              创建 risk_alert（风险提醒）
```

**移除后（简化架构）**：
```
user_subscription → 定时任务检测(11:30/14:30) → 满足条件时创建 risk_alert
```
